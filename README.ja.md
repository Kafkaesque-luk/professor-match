## これは何？(・∀・)

一言でいうと：**研究したいことを中国語（または日本語）の普通の文章で書くと、日本全国の教員から意味的にマッチする人を探して**、三つの段位に分けて出してくれるツール。

大学院受験をした人ならわかるはず。指導教員探しの伝統芸能といえば：

- 大学サイトを開く → 教員ページを 200 件クリック → 30 件目で 3 件目の人を忘れる (´；ω；`)
- やっと運命の先生を見つけた——と思ったら来年定年。研究計画書が無駄に (╯°□°)╯︵ ┻━┻
- あるいは高額なエージェントに頼むと、向こうは Excel で Ctrl+F（実話）

というわけで作りました：あなたは研究したいことを普通の言葉で書くだけ。日本中の教員のふるい分けは機械がやります (•̀ᴗ•́)و

## なんで ChatGPT に聞いちゃダメなの？

いい質問。もちろん LLM に「医用画像 AI を研究している日本の教授は？」と聞けます。返ってくるのは**記憶頼み**の有名どころ 3〜5 人——以上。Web 検索を付けても、読めるのはせいぜい数十ページ、当たった分だけ。

このプロジェクトのやり方は不器用で、だからこそ正確：**約 30 万人分の履歴を事前にベクトル化して、データを全部手元に持つ**。クエリが来たら：

1. **大きく網を打つ**——30 万人と**総当たり**で意味比較、取りこぼしゼロ；
2. **絞って並べる**——決定論的な分野ゲートで畑違いを排除、関連度で並べ、年齢・大学レベルで段位分け。

|  | LLM に直接聞く（Web 検索込み） | Professor Match |
|--|------------------------------|-----------------|
| カバレッジ | 数十ページ、当たり次第 | 約 30 万人**全員をふるいにかける** |
| 見つかる人 | 「記憶にある」有名人だけ | 意味マッチ——無名でもドンピシャな先生が浮上する |
| 真実性 | 人名や研究テーマを捏造しがち | 全結果が実在の履歴 1 行に対応、論文リンクをワンクリックで検証可能 |
| 条件絞り込み | 「33〜55 歳」「非上位 30 校」は無理 | 構造化フィールドでハードフィルタ |
| 手に入るもの | 名前が数個並んだ一段落、チャットが終われば消える | 段位付きリスト + 詳細ページ + 教授と「会話」まで |

要するに：まず全量、それから精密ランキング。データとベクトルを手元に置いているから、この精度が出せます——プロンプトの皮を被せるだけでは届きません。

## 三段位：三つの独立した視点 (￢_￢)

三段位は**同じ候補プールに対する三つの独立した判定基準**。それぞれ別々に判定、重複あり：

| 段位 | ぶっちゃけ |
|------|-----------|
| **ベストマッチ** | 意味的に一番近い、ガチ実力枠 |
| **働き盛り** | 33〜55 歳のマッチ教員。researchmap に誕生日を書く人はいないので、「何年に博士号」「何年に講師就任」から信頼度付きで逆算。「ついた翌年に定年」の悲劇を予防 |
| **狙い目** | 非上位 30 校の強マッチ。競争が緩くて入りやすい、コスパ枠 (¬‿¬) |

## クイックスタート（セルフホスト、API キー不要で起動）

```bash
git clone <your-fork>
cd professor-match
cp .env.example .env

docker compose up -d            # Qdrant + API + Web ターミナル
docker compose run --rm seed    # 同梱の 5,000 教員ベクトルインデックスを投入

# デプロイターミナルを開く：
open http://localhost:8000
```

なぜキーなしで動くのか？サンプル 5,000 人分のベクトルが**計算済みでリポジトリに同梱**されているから。自分のキー（DashScope か OpenAI）が要るのは二つの場合だけ：**新しいクエリ**の埋め込みと、**自前データ**でのインデックス再構築。`.env` かターミナルの「設定」タブで設定できます。

> Docker なしでアルゴリズムだけ見たい？`python scripts/verify_pipeline.py` が検索 → 段位分け → グルーピングの全チェーンをインメモリで一周します。依存ゼロ、キーゼロ ᕕ( ᐛ )ᕗ

## アーキテクチャ：小さくても内臓は本番級

二本の脚、脳はひとつ——Web ターミナルはバックエンド非依存なので、どちらに繋いでも同じ体験：

```text
                      あなた（ブラウザ）
                              │
               ┌──────────────┴───────────────┐
               │  Web ターミナル web/（ビルド不要 SPA）│
               └──────┬────────────────┬──────┘
      セルフホスト経路   │                │    オンラインデモ経路
                      ▼                ▼
        ┌──────────────────┐   ┌─────────────────────────┐
        │  FastAPI（api/）  │   │ 本番サンドボックス端点        │
        │  検索/段位/対話    │   │（読み取り専用 + 三層レート制限 │
        └──┬───────────┬───┘   │ + 自動 BAN + キルスイッチ）  │
           ▼           ▼       └────────────┬────────────┘
      ┌─────────┐ ┌──────────┐              ▼
      │ Qdrant  │ │ DashScope│       本番フルインデックス
      │5000サンプル│ │/OpenAI(任意)│    （約 30 万教員ベクトル
      └─────────┘ └──────────┘     + researchmap 履歴 + Redis）
```

- **検索層**：Qdrant、1024 次元 cosine（DashScope text-embedding-v4）、1 クエリの召回は 150 候補。地域・大学ランク・国公私立は全部ベクトルの payload に同居——**フィルタはストア内で完結、テーブルに戻らない**。
- **データ層**：教員 1 人につき researchmap 形式の履歴 JSON（学歴/職歴/論文/受賞/特許）。年齢は本番では**オフラインで側表に埋め戻し（14 万行超）**、配信時は読むだけ；OSS 版は同じ推定器を同梱してリアルタイム計算。
- **対話層**：五層ペルソナプロンプト（アイデンティティ固定/履歴記憶/キーワードトレンド/研究手法/文体制約 + 項目別の幻覚ガード）、教員ごとに一度組み立ててキャッシュ。**対話は完全ステートレス**——履歴はブラウザが持ち、サーバは何も保存しない。
- **防御層**（オンラインデモ）：三層レート制限（IP/時 · IP/日 · 全体/日）+ 乱用の自動 BAN + fail-closed（キャッシュ障害時は拒否側に倒す）+ 環境変数キルスイッチ。

リクエストが通るパイプライン：

```text
query
  └─▶ キーワード拡張      （LLM;任意 — キーがなければ原文検索に自動降格）
  └─▶ 分野 → cate_id     （決定論的ハードフィルタ;「経済学」で法学は絶対に出ない）
  └─▶ 埋め込み            （DashScope text-embedding-v4 / OpenAI）
  └─▶ Qdrant ベクトル検索 （+ 地域/ランク/国公私立/大学指定フィルタ）
  └─▶ 三段位バケット分け   （関連度 · 33–55 働き盛り · 非上位30 狙い目）
  └─▶ 大学別グルーピング + 統計
```

決定論的コア（年齢推定、分野ゲート、フィルタ変換、段位分け、大学グルーピング）は本番システムからの**一行ずつの忠実移植**で、振る舞いテストが見張っています（`api/tests/test_core.py`）。歴史的クセまでそのまま保存——「本番と同じ味」こそが目的なので。

## 性能とコスト：マッチ 1 回いくら？

小数点以下まで数えるのは、本番が実際にこの請求書を払っているから (´･ω･`)

| ステージ | レイテンシ（実測オーダー） | 限界コスト（DashScope 公開価格で概算） |
|---------|--------------------------|----------------------------------|
| LLM キーワード拡張 | 2–5 秒（チェーンの大半） | 数百トークン、≈ ¥0.002（元） |
| クエリ埋め込み | 200–500 ms | 一文分、≈ ¥0.0001 |
| Qdrant ベクトル検索（約 30 万規模） | < 100 ms | 0（セルフホスト） |
| 段位分け + グルーピング + 年齢注入 | < 50 ms | 0（純 CPU、決定論的） |
| **フルマッチ 1 回** | **≈ 3–10 秒** | **< ¥0.005 元（1 円未満）** |
| AI 対話 1 ターン（qwen-max） | 2–4 秒 | ≈ ¥0.02–0.05 元 |
| 参考：人力の留学エージェントによる教員選び | 往復のやり取りで数日 | 500–1000 元 / 回 |

参考行には現場の裏付けがあります：作者自身が研究計画書の指導と留学出願サポートを本業とし、プロの出願ライターにも聞き取りをしてきました。日常的に使った実感として、このパイプラインが出す教員リストは 1 回 500–1000 元の人力プランニングと同等以上——1 回あたりのコストは 10 万分の 1 です。

コストの大半は LLM で、検索はほぼタダ。だから設計のあちこちで節約しています：

- キーワード拡張は**優雅に降格**——キーがなければ原文で検索、切れ味は少し落ちるが完全無料；
- ペルソナプロンプトは**教員ごとにキャッシュ**——組み立ては一度だけ、DB を読み直さない；
- コーパスのベクトルは**一度計算したら一生もの**——リポジトリには本番オリジナルのベクトルを同梱、クローンした時点で埋め込み代はゼロ；
- オンラインデモは**全体日次上限で最悪ケースを封印**——スクリプトで一晩中叩かれても、請求書はジュース一本分 (´∀｀)

## 精度はどう磨かれたか：チューニングの記録

このパイプラインは一発で正しく書けたものではありません。RAG マッチングを作る人のために、印象深かった調整をいくつか：

**分野汚染。** 「経済学の産業組織」で検索したら文学部と法学部の先生が混入——ベクトルには「組織」「制度」がどこでも似て見える。教訓：**意味検索は漂流する。決定論的なゲートが必須**。分野ハードフィルタを入れてからは、LLM とベクトルはゲートの内側でだけ自由に泳げる。召回が減っても、畑違いは出さない。

**プロンプト乗っ取り。** 以前入力された志望専攻をキーワード拡張に混ぜて「パーソナライズ」したことがある。「幾何学」と登録した人が「電子商取引」を検索したら、画面いっぱいの幾何学用語。修正：**今回の入力がテーマを独占、矛盾する文脈は無視**。プロンプトエンジニアリングの第一課：モデルに渡す文脈は、全部ハンドルを奪いに来る。

**拡張キーワードの数と言語。** 3〜5 個ではニッチ分野の召回が足りず、20 個ではテーマが薄まってランキングが漂う。試行を重ねて 10 個前後に落ち着きました。もうひとつの要点：**必ず日本語に展開すること**——インデックスの言語は日本語なので、中国語キーワードのままだと召回率が崖から落ちます。

**中国語の専攻名のハードマッピング。** 「電子商務」「土木工程」のような中国での呼び方は、日本の学問分類に同名のものがなく、ベクトルは推測するしかない。そこで中国語専攻 → 日本の研究分野の対応表を構築——俗称・略称・簡体繁体まで収録し、ヒットしたら分野を決定論的に固定。この一枚の表が、どのモデルチューニングよりも精度を押し上げました。

**召回ウィンドウ。** 1 クエリで何件引くかも何段階か試しました：50 では少なすぎて「狙い目」段位がよく空になり、300 では尾部のノイズが段位に入り込む。150 で落ち着き、三段位とも満たせて尾部もきれい。

**年齢推定。** 誕生日を書く人はいない。でも「何年に博士号」「何年に講師就任」はある。多シグナル逆算：明示記載 > 学位年 > 初任 + 職位慣例、と信頼度を段階的に下げ、**中〜高信頼度だけが「働き盛り」段位に入れる**。取りこぼしても、間違えない——60 歳の先生を 40 歳として勧めるのは、勧めないより百倍罪深い。

**スコアのキャリブレーション。** 生のコサイン類似度は 0.30–0.62 に密集していて、そのまま見せると「45%」がミスマッチに読める（実際はかなりの好マッチ）。62–97% に単調に引き伸ばして、差がひと目で読めるようにしました。

## Web ターミナル

依存ゼロ・ビルド不要のシングルページコンソールで、**バックエンド非依存**：同じ UI がローカルの Python API にも、（読み取り専用サンドボックス経由で）フルスケールの本番にも繋がります。「設定」タブが接続とランタイム設定、「マッチ」タブが上のスクリーンショットのフルデスクトップ体験——段位タブ、大学別カード、教員詳細ページ（論文最優先 + 履歴タイムライン + 受賞/特許）、AI 対話ドロワー。カード/詳細のパースロジック（`web/professor.js`）はオンライン満血版と完全に同一です。

別のバックエンドに向けるときは `config.js` の `matchPath` / `detailPath` / `chatPath`（`{id}` プレースホルダ）を設定。

**自分のデモを GitHub Pages に公開する：** 同梱のワークフロー（`.github/workflows/pages.yml`）が `web/` を Pages にデプロイします。リポジトリ変数 `DEMO_API_BASE`（任意で `DEMO_MATCH_PATH` / `DEMO_DETAIL_PATH` / `DEMO_CHAT_PATH`）を Settings → Actions → Variables に設定すれば自分のバックエンドに向く；未設定なら同一オリジンのデフォルト——フォークが勝手に他人のバックエンドを叩くことは決してありません。

## 設定

すべて環境変数（`.env`）。重要なのはこれだけ：

| 変数 | デフォルト | 説明 |
|------|-----------|------|
| `QDRANT_URL` | `http://qdrant:6333` | ベクトル DB |
| `EMBEDDING_PROVIDER` | `dashscope` | `dashscope`（同梱インデックスと同源）か `openai` |
| `EMBEDDING_API_KEY` / `QWEN_API_KEY` | — | DashScope キー（新規クエリの埋め込みのみ） |
| `OPENAI_API_KEY` | — | `openai` 利用時（次元が違うので要インデックス再構築） |
| `LLM_PROVIDER` | `dashscope` | キーワード拡張：`dashscope` / `openai` / `none` |
| `DEMO_MODE` | `false` | 公開デモ時に true——admin 設定端点を無効化 |
| `ADMIN_TOKEN` | — | ターミナルのランタイム設定を有効化（ヘッダ `X-Admin-Token`） |

> OpenAI 埋め込みに切り替えると次元が変わり、同梱の DashScope ベクトルと互換性がなくなります。`scripts/build_index.py` で再構築を。

## 自前データを使う

同梱サンプルは本番オリジナルのベクトル（ゼロコスト・即利用可）。自分の教員コーパスに差し替えるには：

1. JSONL で行を用意（`product_id`、`store_name`、`extend`、`school_rank`、`school_name`、`school_type`、`school_region_id`、`cate_id`、`image`）。`extend` は researchmap 形式の履歴 JSON。
2. `python scripts/build_index.py --rows data/your_rows.jsonl`（埋め込みキーが必要;再埋め込みします）。

`scripts/export_from_prod.py` は CRMEB 系 MySQL + Qdrant から SSH 経由でサンプルを抜く読み取り専用のリファレンスエクスポータ（`PM_*` 環境変数で設定）。

## データとプライバシー

サンプルデータは公開されている [researchmap](https://researchmap.jp) の学術プロフィール（氏名・所属・論文リンク）由来で、**連絡先なし・業務データなし**。AI 対話は完全ステートレス：履歴はブラウザが持ち、サーバは 1 バイトも保存しません。気になる場合は、自分のエクスポートを公開する前に匿名化を。
